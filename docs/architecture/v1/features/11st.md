## 0. 배경 — 왜 재구현인가

## 1. 확정 결정 (사용자 승인 — 변경 금지)

1. **spec 기준 전면 재작성** — 5메서드(authenticate 제외 4 + 주문/발송 2)를 `11st-api/` 문서 endpoint·필드로 재작성. authenticate(API Key 저장 + category ping)·credential 저장 경로는 정상이므로 유지.
2. **범위 = 상품등록 + 주문/배송** (PRD §1·§6). 클레임(취소/교환/반품 48~50)·정산(151)·기획전(54)·알리미(58)·해외물류/전세계배송(63~69)은 v2.
3. **배송 2-레이어 준수** (`shipping-fee-model.md`):
   - **Layer 1(요금 의도)** = `shipping_policies` → `transformProduct` 가 11번가 인라인 필드(`dlvCstInstBasiCd`/`dlvCst1`/`PrdFrDlvBasiAmt`/`jejuDlvCst`/`islandDlvCst`/`rtngdDlvCst`/`exchDlvCst`/`bndlDlvCnYn`)로 매핑.
   - **Layer 2(물류 참조)** = 출고지/반품지 `addrSeq`. **11번가는 ESM 과 달리 우리가 생성하지 않는다** — 셀러가 11번가 셀러오피스에서 선행 등록한 출고지/반품지를 **조회 API(1014/1691·1015/1692)로 fetch → 셀러가 select**. (등록 API 6701/6702 는 v2 편의 기능.)
4. **상품정보고시(`ProductNotification`)** 는 s3 등록 3단계 카드에 입력. ESM PR-5 의 `officialNotice` 프레임워크(`getRegistrationFields` `kind:'officialNotice'`)를 **재사용**하되, 11번가 상품군 코드(`type`)·항목(`item.code`) 마스터는 별도(11번가 spec 기준).
5. **마켓별 동적 등록필드** — ESM 이 도입한 `getRegistrationFields()`(`market-adapter.md` §9.8)를 11번가도 구현: `[출고지 select, 반품지 select, officialNotice]`. 컴포넌트 `if (marketId==='11st')` 하드코딩 금지.
6. **real 검증은 mock + parity 까지** (ESM 과 동일). 실제 `api.11st.co.kr` 실호출 검증은 셀러 발급 키 + IP 화이트리스트(`3.36.239.243`) 확보 후 별도 라운드(WIP C3 계열).

## 3. Layer 2 데이터 모델 — 출고지/반품지 참조

11번가는 출고지/반품지를 **우리가 생성하지 않는다** (ESM 의 `esm_shipping_profiles` 4단계 생성과 핵심 차이). 셀러오피스 선행 등록분을 조회·선택만 한다. 두 가지 보관 전략:

- **조회형 (확정)** — 캐시 테이블 없이 **조회 API 를 호출 시점에 fetch**(24h 캐시는 호출측). 상품등록 3단계 select 옵션 = `outboundarea`(1014) / 반품지(1015) 응답의 `addrSeq`+`addrNm`. 등록 payload 엔 셀러가 고른 `addrSeq` 만 주입. 마이그레이션 0, 동기화 문제 없음.

> **결정 (2026-05-30, B안)**: 11번가뿐 아니라 **5마켓 전부 조회형 단일 표준**, 배송 참조 DB 테이블 없음(`shipping-fee-model.md §2`). ESM 의 생성형 테이블(`esm_shipping_profiles`)은 deprecate(`esm.md` 전환 결정 절). 범용 테이블안은 폐기.

조회 응답 (spec 확정):

| API | endpoint | 응답 root | 핵심 필드 |
|---|---|---|---|
| 출고지 목록 | `GET /rest/areaservice/outboundarea` (1014) | `ns2:inOutAddresss > ns2:inOutAddress[]` | `addrSeq`(시퀀스)·`addrNm`(주소명)·`addr`·`rcvrNm`·`gnrlTlphnNo`/`prtblTlphnNo` |
| 출고지 단건 | `POST /rest/areaservice/getOutAddressInfo/{addrSeq}` (1691) | 동일 | + `dtlsAddr`·`mailNO`·`buildMngNO` |
| 반품/교환지 | (1015 조회 / 1692 단건) | 동일 | `addrSeq`·`addrNm` |

- `result_message == 'SUCCESS'` 체크 후 `addrSeq` 사용 (spec 명시). `ns2:` prefix 제거 필요.
- PII(주소·전화)는 **우리 DB 에 저장 금지** — select 표시는 `addrNm`(주소명)만, payload 는 `addrSeq` 만.

### 4.2 상품등록 응답 (`ElevenStProductCreateResponseSchema`)

문서 `product-manage-1003.md` 응답. root **`ClientMessage`** (구 코드 `Product>ProductNo` 는 오기):

| 경로 | 타입 | 의미 |
|---|---|---|
| `ClientMessage.productNo` | string | 11번가 상품번호 → `externalId` |
| `ClientMessage.resultCode` | enum | **200=일반성공 / 210=신규성공** / 400=일 500개 한도 / 500=실패 |
| `ClientMessage.message` | string | 결과 메시지(에러 전부 메시지로) |

- 성공 판정 = `resultCode ∈ {200, 210}` **AND** `productNo` 존재. 그 외 `MarketError`(코드별: 400=rate-ish/validation, 500=validation, 메시지 동봉).
- `productUrl` = `https://www.11st.co.kr/products/{productNo}` (현 코드 유지).

### 4.3 카테고리 응답 (`ElevenStCategorySchema`)

|---|---|
| `dispNo` | `id` |
| `dispNm` | `name` |
| `depth` | `depth`(1대/2중/3소/4세) |
| `parentDispNo` | `parentId`(0=최상위→null) |

## 8. cross-market 공통 처리 (사용자 요청 — 11번가가 확정시킨 보편 패턴)

11번가 spec 대조로 "마켓 1개 특수" 가 아니라 **마켓 공통 관심사**임이 드러난 4건:

1. **출고지/반품지 사전 참조 (Layer 2 — 조회형 단일 표준)** — 5마켓 전부 출고지/반품지를 사전 리소스로 참조함이 확정: 쿠팡 `outboundShippingPlaceCode`/`returnCenterCode` · 네이버 `shippingAddressId`/`returnAddressId` · ESM `placeNo`/`dispatchPolicyNo` · **11번가 `addrSeq`**. **5마켓 모두 "셀러가 마켓 콘솔에서 만든 리소스를 우리가 조회 → select" 하는 조회형으로 통일한다**(`shipping-fee-model.md §2` Layer 2). `getRegistrationFields`(§9.8) select 의 `optionsSource` = 마켓별 lookup. 배송 참조 DB 테이블 없음. ⚠️ ESM 은 현재 생성형(우리가 4단계 생성)으로 구현돼 있으나 조회형으로 **전환**한다(생성형은 API 강제 아님 — `esm.md` "전환 결정" 절). **생성형을 신규 마켓 템플릿으로 삼지 않는다.**
2. **상품정보고시 (officialNotice 보편화)** — 11번가 `ProductNotification`(`type`+`item`), ESM `officialNotice`(41 상품군) 동형. `getRegistrationFields` `kind:'officialNotice'` + `OfficialNoticeField` UI 를 공통 프레임으로 유지하고, **마켓별 상품군 코드 마스터만 분리**(`ESM_OFFICIAL_NOTICE_GROUPS` / `ELEVEN_ST_NOTICE_GROUPS`). 쿠팡/네이버도 동일 고시 요구 → 후속 흡수.
3. **택배사 코드 매핑** ✅ **(PR-6 구현 완료)** — 11번가 `dlvEtprsCd`(00034=CJ/00002=로젠/00011=한진/00012=롯데/00007=우체국…), ESM(`DeliveryCompanyCode` 숫자) 코드 상이. PR-5 까지 11번가(`ELEVEN_ST_CARRIER_CODES`)·ESM(`ESM_CARRIER_CODE`)이 각자 내부맵을 들고 있었음 → **공통 `CarrierCode`(내부 enum, `CARRIER_CODES` 10종/default LOGEN) → 마켓별 코드 매핑 테이블** 단일 소스 `_shared/carrier-codes.ts`(+ Web `lib/markets/carrier-codes.ts` 미러)로 추출. 변환 함수(`toElevenStCarrierCode`/`toEsmCarrierCode`)·정규화(`normalizeCarrierCode`) 제공. 11번가·ESM 어댑터(Web+Edge) 참조만(기존 export 는 map re-export 호환). 미매핑은 undefined → 호출측 `unsupported_carrier`(코드 날조 금지). 쿠팡/네이버는 영문 코드(`'LOGEN'`) 그대로 전달이라 현 단계 매핑 불요 — v2 다중 택배사 진입 시 본 모듈에 흡수.
4. **XML + ns2 네임스페이스 파서** ✅ **(PR-6 구현 완료)** — 11번가만 XML/EUC-KR + 응답 `ns2:` prefix. `stripNsPrefix`(재귀 prefix 제거)를 공용 유틸 `_shared/xml.ts`(+ Web `lib/markets/xml.ts` 미러)로 추출(EUC-KR 디코딩은 fetch 본체 어댑터에 잔존 — 순수 파서만 공통화). 11번가 map(Web+Edge)·`eleven-st-shipping-list/lib/normalize.ts` 가 re-export 경유 사용. ESM(JSON)·쿠팡/네이버(JSON)와 공유는 적으나 11번가 내부 일관성 + 향후 XML 마켓 대비.

> §8 은 PR-6 으로 묶되, 1·2 의 프레임 재사용은 PR-2/PR-4 에서 자연 발생. 택배사 맵(3)은 PR-5 선행 필요. **(2026-05-30 PR-6: 3·4 추출 완료. 1·2 추가 일반화는 쿠팡/네이버 출고지·고시 조달과 함께 v2 후속 — 과잉 추상화 방지.)**

## 9. 범위 경계 & 후속(backlog)

**v1 재구현 범위**: 카테고리·상품등록(단일상품)·출고지/반품지 select·상품정보고시·주문조회·발송처리 + §8 공통화.

**v2 / 후속**:
- 멀티옵션(`ProductOption`/`ProductRootOption`/추가구성 `ProductComponent`) — v1 단일상품만.
- 클레임(취소/교환/반품 48~50)·정산(151)·기획전(54)·알리미(58)·해외물류/전세계배송(63~69).
- 출고지/반품지 **등록** API(6701/6702) — v1 은 조회만.
- **11번가 상품군(officialNotice) 코드 마스터 출처 (C4 — PR-4 에서 부분 해소)** — spec 1003 의 `ProductNotification.type`/`item.code` 가 외부 첨부파일 참조라 우리 docs(`11st-api/`)에 41군 전체 코드가 없다. **PR-4 확보 범위 = spec 1003 예시 1군(`type='891011'`, 예시 item `23759468`/`나일론`)뿐** → `ELEVEN_ST_NOTICE_GROUPS` 에 1군만 정적(`docVerified:true`) 박고, 나머지 40군은 **셀러 free-form**(`OfficialNoticeField` 의 "직접 입력" 옵션 — 추측 코드 날조 금지)으로 처리. 후속: 11번가 첨부파일(상품군 코드표) 또는 고시 조회 API 확보 시 `ELEVEN_ST_NOTICE_GROUPS` 보강(ESM 라이브 API 방식과 동형). 군별 정적 필수항목(`requiredItemCodes`)도 첨부파일 의존이라 v1 미설정.
- Layer 2 범용 테이블(B안) — 쿠팡/네이버 출고지·반품지 조달과 함께.
- KC인증(`ProductCertGroup`) 항목별 입력 UI 정교화 — v1 은 카테고리 `requiredYn` 시 최소 입력.
