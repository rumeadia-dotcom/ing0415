# Git 커밋 자동화

이 커맨드는 Git 변경사항을 분석하여 적절한 커밋 메시지를 자동 생성하고, 사용자 확인 후 커밋을 수행합니다.

## 실행 절차

1. **Git 저장소 확인**
   - 현재 디렉토리가 Git 저장소인지 확인합니다.
   - Git 저장소가 아니면 오류 메시지를 표시하고 종료합니다.

2. **변경사항 확인**
   - `git status` 명령어를 실행하여 변경된 파일 목록을 확인합니다.
   - 변경사항이 없으면 "커밋할 변경사항이 없습니다" 메시지를 표시하고 종료합니다.

3. **변경 내용 분석**
   - `git diff` 명령어를 실행하여 변경된 내용을 상세히 분석합니다.
   - 변경된 파일들의 내용을 읽어서 어떤 수정이 이루어졌는지 파악합니다.
   - 추가된 파일, 수정된 파일, 삭제된 파일을 구분하여 분석합니다.
   - 각 파일의 변경 내용을 분석하여 주요 변경사항을 파악합니다.

4. **커밋 분리 판단**
   - 변경 내용 분석 결과, **기능 단위가 서로 다른** 변경사항이 섞여 있으면 **분리 커밋을 제안**합니다.
   - 분리 기준 (하나 이상 해당 시 분리 제안):
     - **기능 단위가 다름**: 백엔드/프론트엔드, 서로 무관한 두 기능, 도구·설정과 제품 코드 등 독립적 단위
     - **의존성 방향**: A가 B에 의존하면 B를 먼저 커밋 (예: 백엔드 → 프론트엔드)
   - **type 차이 단독으로는 분리하지 않습니다**: 같은 기능 단위의 코드·문서·테스트·설정 변경은 함께 묶는 것이 자연스럽습니다. 예시:
     - `feat`(코드) + `docs`(같은 기능의 인수인계/README) → 묶음
     - `feat`(API) + `test`(그 API 의 테스트) → 묶음
     - `feat`(기능 A) + `chore`(무관한 도구 추가) → 분리 (기능 단위 다름)
   - 묶인 커밋의 `<type>` 은 묶음의 **주된 변경**을 따릅니다. 예: 코드 90% + 문서 10% → `feat`.
   - 분리가 필요하면 다음 형식으로 제안합니다:
     ```
     커밋을 N개로 분리하는 것을 제안합니다:

     1. <type>: <subject> — <포함 파일 요약>
     2. <type>: <subject> — <포함 파일 요약>
     ...
     ```
   - 분리 순서 원칙:
     - 의존성 순서 (의존 당하는 쪽 먼저)
     - 핵심 코드 → 문서/설정 → 메타 정보
   - 사용자가 분리를 승인하면 각 커밋을 순서대로 진행합니다 (커밋별로 5~7단계 반복).
   - 사용자가 단일 커밋을 원하면 그대로 진행합니다.

5. **커밋 메시지 생성**
   - 변경사항을 분석하여 적절한 커밋 메시지를 생성합니다.
   - 커밋 메시지는 프로젝트의 README.md에 명시된 컨벤션을 엄격히 따라야 합니다:
     - 형식: `<type>: <subject>` (빈 줄) `<body>`
     - type 종류: feat, fix, docs, style, refactor, test, chore, perf, ci
       - 변경사항의 성격에 맞는 type을 선택합니다.
       - 새로운 기능 추가 → feat
       - 버그 수정 → fix
       - 문서 수정 → docs
       - 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음) → style
       - 코드 리팩토링 → refactor
       - 테스트 코드 추가/수정 → test
       - 빌드 업무 수정, 패키지 매니저 설정 등 → chore
       - 성능 개선 → perf
       - 빌드 시스템 또는 외부 종속성에 영향을 미치는 변경 사항 → build
       - CI/CD 설정 변경 → ci
     - subject: 50자 이내, 명령형, 과거형 사용 안함, 마침표 사용 안함
       - 예: "사용자 인증 기능 추가" (O), "사용자 인증 기능을 추가했습니다" (X)
     - body: 무엇과 왜를 설명 (옵션, 변경사항이 복잡한 경우 작성)
       - 변경사항의 핵심 내용을 명확히 설명합니다.
       - 왜 이 변경이 필요한지 이유를 설명합니다.
       - 구현 방법은 생략하고 무엇을 했는지, 왜 했는지만 설명합니다.
   - 개발 내용 및 수정 사항을 참고하여 요점을 명확히 제시합니다.
   - git diff의 내용을 자세히 분석하여 정확한 커밋 메시지를 생성합니다.
   - 한국어로 작성합니다.
   - **커밋 메시지에 "🤖 Generated with Claude Code" 등 Claude 관련 프리픽스나 서픽스를 포함하지 않습니다.**
   - README.md의 예제를 참고하여 유사한 형식으로 작성합니다:
     ```
     feat: 사용자 인증 기능 추가

     사용자 인증 코드 생성을 위한 HASH 알고리즘으로, SHA-256 사용한다. 인증 코드 자체는 DB에 저장하지 않는다.
     이는 보안 정책에 의한 것이며, 인증 코드 비교를 위해 매번 생성한다.
     해당 인증 코드는 유효 시간을 가지며, 유효 시간이 지난 인증 코드 유입 시 인증 실패 처리한다.
     ```

6. **사용자 확인** (분리 커밋 시 각 커밋마다 반복)
   - 생성된 커밋 메시지를 사용자에게 명확하게 제시합니다.
   - 커밋 메시지를 다음과 같은 형식으로 표시합니다:
     ```
     생성된 커밋 메시지:

     <type>: <subject>

     <body>
     ```
   - 사용자가 메시지를 수정하거나 확인할 수 있도록 대화형으로 진행합니다.
   - 사용자가 메시지를 수정하고 싶으면 수정된 내용을 입력받습니다.
   - 사용자가 "확인", "커밋", "예", "y", "yes" 등을 선택하면 다음 단계로 진행합니다.
   - 사용자가 "취소", "아니오", "n", "no" 등을 선택하면 커밋을 취소합니다.

7. **커밋 실행** (분리 커밋 시 각 커밋마다 반복)
   - 스테이징 전, untracked/modified 파일 중 `.gitignore`에 추가해야 할 파일이 있는지 검토합니다:
     - 민감 정보: `.env`, 인증서, 키 파일, credentials 등
     - 빌드 산출물: `dist/`, `build/`, `*.pyc`, `node_modules/` 등
     - 사용자별 설정: IDE 설정, 로컬 설정 파일 등
   - 주의가 필요한 파일이 발견되면 커밋 전에 경고하고, `.gitignore` 추가 여부를 사용자에게 확인합니다.
   - 사용자 확인 후 변경사항을 스테이징합니다. (분리 커밋 시 해당 그룹 파일만 `git add <파일목록>`)
   - 확인된 커밋 메시지로 `git commit` 명령어를 실행합니다.
   - 커밋 메시지 형식에 따라 적절한 방법을 사용합니다:
     - subject만 있는 경우: `git commit -m "<type>: <subject>"`
     - body가 있는 경우: HEREDOC을 사용하여 커밋 메시지 작성
   - 커밋이 성공적으로 완료되면 성공 메시지를 표시합니다.
   - 커밋 실패 시 오류 메시지를 표시하고 원인을 설명합니다.
   - 푸시는 수행하지 않습니다 (사용자가 직접 수행해야 함).

8. **태그 추가 (선택)** (분리 커밋 시 마지막 커밋 완료 후 1회만)
   - 커밋이 성공적으로 완료된 후, 사용자에게 태그를 추가할지 질문합니다.
   - 사용자가 태그 추가를 원하면 다음을 진행합니다:
     - 기존 태그 목록을 `git tag --sort=-creatordate | head -5`로 조회하여 최근 태그를 참고용으로 표시합니다.
     - 변경사항의 성격과 기존 태그 네이밍 패턴을 분석하여 태그 이름을 제안합니다.
       - 기존 태그가 있으면 그 패턴(예: `v1.0.0`, `release-1.0`)을 따릅니다.
       - 기존 태그가 없으면 Semantic Versioning (`v0.1.0`) 형식을 제안합니다.
     - 태그 메시지는 커밋 메시지의 subject를 기본값으로 제안합니다.
   - **사용자 확인**: 태그 이름과 메시지를 다음 형식으로 표시하고 확인을 받습니다:
     ```
     태그 정보:
     - 태그: <tag_name>
     - 메시지: <tag_message>
     ```
   - 사용자가 수정을 원하면 수정된 내용을 입력받습니다.
   - 사용자가 확인하면 태그 생성과 푸시를 순서대로 실행합니다:
     1. `git tag -a <tag_name> -m "<tag_message>"` — 태그 생성
     2. `git push origin <tag_name>` — 원격 저장소에 태그 푸시
   - 태그 생성 및 푸시가 완료되면 성공 메시지를 표시합니다.
   - 사용자가 태그 추가를 원하지 않으면 이 단계를 건너뜁니다.

## 에러 처리

- Git 저장소가 아닌 경우: 명확한 오류 메시지 표시
- 변경사항이 없는 경우: 안내 메시지 표시 후 종료
- Git 명령어 실행 실패 시: 오류 메시지 표시 및 원인 설명
- 이미 존재하는 태그 이름을 입력한 경우: 안내 메시지 표시 후 다른 이름 입력 요청

## 주의사항

- 커밋은 원격 저장소로 푸시하지 않습니다 (사용자가 직접 수행).
- 태그는 사용자 확인 후 생성과 동시에 원격 저장소로 푸시합니다.
- 사용자가 최종 확인을 거쳐야 커밋이 실행됩니다.
- 사용자가 최종 확인을 거쳐야 태그가 생성됩니다.
- 커밋 메시지는 프로젝트의 커밋 컨벤션을 준수해야 합니다.
- 커밋 메시지에 Claude 관련 프리픽스/서픽스를 포함하지 않습니다.
